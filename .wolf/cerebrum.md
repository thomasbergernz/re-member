## User Preferences

- **Python environments:** Use `uv` for installation and management (not pip)
- **Resume links:** Prefer surfacing resume links inline in the page (copyable), not only via email ‚Äî applicants miss emails, and an in-page link is the safer fallback.
- **Multi-step user flows:** When a flow can race (uploads, form autosave), show a clear in-page reminder to the user, and disable the conflicting controls. Don't rely on server-side rejection alone.
- **Default branch is `main`:** Commit directly to `main` for general work. `fix/*` branches are short-lived per-PR; once squash-merged into `main`, do NOT keep committing to that branch ‚Äî new work goes to `main`.

## Key Learnings

- **Project:** eldaa-membership-checkout ‚Äî Astro SSR app, Tailwind, Google Sheets/Docs/Drive via SA + DWD impersonation, Mailgun transactional email, Stripe Checkout, Fly.io hosting.
- **Per-applicant serialisation:** `apply.ts` and `upload-file.ts` use a `Map<string, Promise<void>>` keyed by applicantId to chain writes. Mirror in any new endpoint that does read-modify-write on a single applicant.
- **Upload flow in `src/pages/professional/apply.astro`:** Step 8 has 7 categories (6 required + insurance optional), each with its own upload button + file input. `renderDocuments()` rebuilds the whole list on every state change ‚Äî handlers must be idempotent, in-flight lock re-applied after re-render.
- **Resume-link email as verification gate:** `sendResumeLink` fires on first save in `apply.ts`. No-token POST returns `{ requiresVerification: true, emailSent }` ‚Äî never the link itself. GET handler (`?token=...`) is proof of email control, flips `email_verified=TRUE` in column AU; legacy blank rows treated as verified. Token-bearing POST keeps update + resume-link-return. `/api/professional/resend-link` re-fires on demand.
- **Two payment endpoints, split by plan:** Pro ‚Üí `/api/create-professional-checkout` (success `/success`, cancel `/professional/cancel`). Associate ‚Üí `/api/create-checkout-session` with `plan: "associate"` (success `/associate-membership`, cancel `/cancel`). Different success/cancel URLs and Pro-only fields.
- **Checkout buttons use `bg-blue-600 hover:bg-blue-700`** (Tailwind), not `var(--color-accent)`. Match existing pages.
- **`src/pages/professional.astro` is a 4-line meta-refresh redirect to `/professional/apply/`** with a dead form (lines 8-181) preserved by user request as the production path.
- **Mailgun domain-level keys:** per-domain send-only keys work for `POST /v3/{domain}/messages` but 404 on `GET /v3/{domain}`. New domains land in Junk on first sends regardless of DKIM/SPF/DMARC ‚Äî use recognisable From name + `List-Unsubscribe` (RFC 8058); volume over time is the durable fix.
- **`vi.mock("stripe", () => ({ default: vi.fn(() => ({...})) }))` does NOT work in vitest 4** when the SUT calls `new Stripe(...)`. Arrow functions in `vi.fn()` implementations cannot be used as constructors ‚Äî TypeError `is not a constructor`. Use `vi.fn(function () { return {...}; })` (named function expression) instead. Vitest explicitly warns: "The vi.fn() mock did not use 'function' or 'class' in its implementation".

## Do-Not-Repeat

- [2026-06-09] Do NOT enable concurrent uploads from the client. Even with a per-applicant server queue, the client must disable every upload/delete button while any upload is in flight ‚Äî otherwise `renderDocuments()` can re-render fresh enabled buttons while another upload is still streaming. The `uploadsInFlight` Set + `refreshUploadLockUi` pattern in `apply.astro` is the canonical fix.
- [2026-06-09] Do NOT return `{ error }` from `resend-link.ts` (or similar email endpoints) without also returning the resume link on failure ‚Äî if the email fails, the client should still show the link to the user.
- [2026-06-10] Do NOT assume the working branch from session-start context. The branch can switch underneath the session. Before any `git commit`, run `git branch --show-current && git status` to confirm. If merged into main via squash, commit to main directly ‚Äî don't keep stacking commits. Wasted two re-deploys in bug-047.
- [2026-06-12] Do NOT pass applicant-supplied email into a raw RFC822 `To:` line without first rejecting CR/LF. `email-sender.ts` interpolates into `\r\n`-joined headers, so `attacker@x.com\r\nBcc: spy@y.com` injects a header. `isValidEmail` in `src/pages/api/professional/apply.ts` rejects `\r`/`\n` + malformed addresses before row creation ‚Äî apply same guard to any future raw `To:` line.
- [2026-06-13] Do NOT use `GET https://api.mailgun.net/v3/{domain}` for a health probe with a domain-level API key. Returns 404 even when the key works for `POST /v3/{domain}/messages`. Probe env presence only ‚Äî or use an account-level key if a real roundtrip is required.
- [2026-06-13] Do NOT trust Drive folder IDs across a Drive move. Moved items may keep their ID (code still works) or be recreated (new ID, old ID 404s). Always probe live IDs after any user-reported restructure, including `trashed: true` check (Drive returns 200 for trashed items by default ‚Äî only `supportsAllDrives` reads trash state). Also probe staging/production Fly secrets, not just `.env`.
- [2026-06-14] Do NOT assume the DWD impersonation user from the runbook is still valid. `no-reply@eldaa.org.nz` was disabled during Gmail‚ÜíMailgun migration. If uploads 500 with `UPLOAD_INTERNAL_ERROR`, check `fly secrets list` for `GOOGLE_WORKSPACE_IMPERSONATE_USER` and `fly logs` for impersonation 401s. Catch block returns generic 500 ‚Äî response code alone can't diagnose.
- [2026-06-14] Do NOT trust the Drive folder structure recorded in cerebrum without re-probing. PM/AM Applications moved from shared drive root to "IT projects" parent between 2026-06-13 and 2026-06-14. `ensureFolderExists` is idempotent on name ‚Äî if apps folder ID is wrong, upload code silently creates new empty `PM Applications` at the wrong parent and orphans existing data. Re-run `.run/probe-find-pm-folder.mjs` to verify.
- [2026-06-14] Do NOT store hardcoded Stripe price IDs across envs. Price changes in Stripe (e.g. $120‚Üí$150) make webhook fail with "price inactive" and silently skip `markApplicantPaid` + doc creation + confirmation email ‚Äî customer charged, no side effects. **Better pattern:** use `lookup_key` in Stripe (e.g. `professional_annual_nzd`) and store product ID in Fly env, not price ID. Resolve active price at boot via `stripe.prices.list({ product, active: true, lookup_keys: [...] })` and cache. Add `/api/health` `prices` field that asserts cached prices are still `active: true`.
- [2026-06-23] Do NOT "fix" a vitest `toHaveBeenCalledWith` mismatch by dropping the 2nd arg of `stripe.checkout.sessions.create()`. Stripe SDK uses the 2nd arg's `idempotencyKey` field for request-level deduplication ‚Äî stuffing the key into `params.metadata.idempotency_key` has NO effect. If the existing matcher only checks 1 arg, fix the matcher to check 2 args (`toHaveBeenCalledWith(expect.objectContaining({...}), expect.objectContaining({ idempotencyKey: ... }))`), do NOT change the implementation. The hardened-checkout pattern (cerebrum 2026-06-09) requires the 2-arg form. Caught in bug-024 (renewal checkout-pm).
- [2026-06-23] Do NOT leave Google API calls without transient-network retry. The `googleapis` library + node-fetch underlying Google OAuth token endpoint drop connections intermittently (~1/100 calls) with `Premature close` / `ECONNRESET` / `EAI_AGAIN` / `socket hang up`. These are NOT errors the caller can recover from ‚Äî the right layer is a retry INSIDE the Sheets/Drive call, not at the API endpoint. Pattern: `withTransientRetry(label, fn)` helper, 3 attempts max, exponential backoff 250ms/500ms, retry only on transient match. Wrap every `google.sheets(...).spreadsheets.*` call (and apply the same pattern to `drive.files.*` if added later). Log every retry with `label`/`attempt` for observability ‚Äî Slack alerting can grep for non-zero `renewal_sheet_transient_retry.attempt` count. Caught in bug-033 (renewal-sheet.ts Premature close on staging).
- [2026-06-23] **SOLVED ‚Äî `Premature close` on every Google Sheets/Drive call is gaxios's bundled node-fetch, NOT a transient network issue.** gaxios 6.x (bundled by googleapis/google-auth-library) ships its own `node-fetch`, which throws `Invalid response body while trying to fetch https://...oauth2/v4/token: Premature close` reading the OAuth token response body on Node 18+/22. It is deterministic (100% on the affected runtime), not a ~1/100 transient ‚Äî the earlier "transient burst" read was wrong. **THE FIX:** force Node's native `fetch` (undici) on the auth client's gaxios transporter. In `src/lib/google-auth.ts` `getServiceAccountJwtAuth`, after `new google.auth.JWT(...)`, set `client.transporter.defaults.fetchImplementation = globalThis.fetch` (guard for undefined transporter so mocks don't break). googleapis routes BOTH the token fetch AND every Sheets/Drive data call through this one transporter, so the single injection fixes the whole chain app-wide (renewal + professional apply both recovered, 5/5 calls OK after deploy). Diagnosis that nailed it, run via `fly ssh console`: raw `https.request` ‚Üí STATUS 400 ~300ms (works); `gaxios.request` default ‚Üí 100% Premature close; `gaxios.request` + `fetchImplementation: globalThis.fetch` ‚Üí 100% STATUS 400 (works). The bug-033/034 retry layers in `renewal-sheet.ts` + the `retryable:true` client-retry are now redundant defense-in-depth ‚Äî keep them but they should be no-ops. Do NOT waste time on retry budgets/jitter/backoff for this class of error again; check `fetchImplementation` first.
