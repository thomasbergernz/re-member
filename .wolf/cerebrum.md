## Key Learnings

- **Astro component self-import is not supported.** Recursive components need a sibling helper file (e.g. `InlineRenderer.astro` that wraps `FieldRenderer.astro`) — Astro refuses `import FieldRenderer from "./FieldRenderer.astro"` inside the same file when it would loop.
- **Schema-validation test for option-value references** is load-bearing: any future change that moves option values into JSON will fail `assertOptionValuesExist` at first call. Plan review finding M3.
- **Astro server route `src/pages/api/renew/checkout/[tier].ts`** uses SSR mode (`output: "server"` + `@astrojs/node` standalone). No `getStaticPaths` needed — params resolve at request time. Adding one would force prerender and break the route.
- **`getRenewalById` reader** (`renewal-sheet.ts:331`) hardcodes `match[1] === "am" ? "am" : "pm"` — defaults any unknown tier to `"pm"`. Phase A config-only path cannot add a third tier cleanly until this reader is made data-driven (Phase D).
- **Playwright CANNOT intercept server-side Stripe/Sheets/Mailgun calls** — they are Node calls inside Astro API routes, not browser requests. E2E neutralises them via env-gated server shims, NOT `page.route()`: `E2E_STUB=1` makes `getSheetsClient()` (`upload-sheet.ts`) return an in-memory client and `sendEmail()` (`email-sender.ts`) return success; a recipient containing `forcefail` throws so ONE built server drives both the email-success and email-failure paths. `GOOGLE_SHEETS_SPREADSHEET_ID` must still be set non-empty (helpers assert presence before reaching the stub). Shims are no-ops when the env vars are unset.
- **CI now gates on tests.** `.github/workflows/test.yml` runs `npm run check` + `npm test` + `npm run test:e2e` on every PR + push to main. The `fly-deploy*.yml` workflows still run NO tests — they only build/deploy. bug-004/bug-005 shipped precisely because nothing ran the 285-test suite before deploy.
- **Stale-path guard** (`src/lib/__guards__/stale-paths.test.ts`) greps all `src/**/*.{astro,ts}` for `/professional/` route literals and asserts every `/api/` path the apply pages fetch maps to a real route file. It excludes its own file. The grep guard catches non-UI literals (logger strings, Stripe redirect URLs) the Playwright smoke can't reach without a completed payment.

## Do-Not-Repeat

- **(2026-06-26)** When defining a `repeatable` FieldDefinition in tests/schemas, ALWAYS include `itemFields`. The TypeScript type requires it; missing it surfaces as `ts(2322)` in astro check.
- **(2026-06-26)** Don't put backticks inside template literals — esbuild transform fails with `Expected ")" but found "schema"`. Use plain words or escape: ``${id} — no schema export in ${tsPath}`` not ``${id} — no `schema` export``.
- **(2026-06-26)** `ynRadio` validator semantics: blank is invalid (not optional). The presence of "yes" or "no" is required by definition; if blank should be allowed, add `required: false` on the field and let the `required` validator decide. Earlier draft had blank-skipped logic which was wrong.
- **(2026-06-27)** TierConfig.storageValue vs RenewalInput.tier type mismatch: `storageValue: string` (flexible) vs `tier: "pm" | "am"` (literal). Cast at the boundary: `tierConfig.storageValue as "pm" | "am"` until Phase D tightens TierConfig. Without the cast: `ts(2322)` at the call site.
- **(2026-06-29)** `is:inline define:vars={{ ... }}` only injects listed variables. Astro server-side imports used as raw JS identifiers inside inline IIFEs MUST be added to `define:vars`. A missing variable causes a silent `ReferenceError` that aborts the entire IIFE — any overlay/spinner-hide code later in the script will never run. Pattern: if using `someServerVar` in IIFE body, add it: `define:vars={{ token, someServerVar }}`.
- **(2026-06-29)** A `professional → advanced` (or any route) rename must sweep MORE than top-level fetch paths: `.astro` inline-script string literals, API redirect URLs (Stripe `success_url`/`cancel_url`), `<a href>` in cancel/landing pages, and `logger.child({ route })` strings ALL hold route literals. bug-004 fixed only apply.astro + 2 handlers; bug-006 was the residue (upload-complete.ts redirects, cancel.astro, cancel-upload.astro, upload-file.ts logger). The grep guard now enforces zero residue.
- **(2026-06-29)** Do NOT `git checkout <file>` to "undo" a temporary edit when that file ALSO has earlier uncommitted changes you want to keep — checkout reverts to the committed (pre-fix) state and silently wipes the good edits. During the guard fails-on-regression sanity check this erased the cancel-upload.astro `/advanced/` fix; it had to be re-applied. Use a scratch copy or `git stash` instead, or edit-then-edit-back rather than checkout.
- **(2026-06-28)** FieldRenderer + Step emit class names (`.field`, `.field-label`, `.field-help`, `.form-step`, `.step-title`, `.radio-group`, `.radio-option`, `.checkbox-label`, `.field-group`, `.repeatable`, `.field-grid`) that must be styled in `src/styles/global.css`. Without them, schema-driven pages render as the Image #10 "messy" view (unstyled default browser form). The CSS lives in `@layer components` and mirrors the inline Tailwind classes used by `src/pages/advanced/apply.astro` registration section. Any new FieldRenderer variant (e.g. `date` added later) needs both an `.field input[type="X"]` selector AND a `grid-column: 1 / -1` rule if it should span both columns.

## Key Learnings

- **(2026-06-27)** Phase F: `validate()` honours `field.required` regardless of `validators[]` content. Skipped only when validators[] contains one of `required`/`conditional`/`ynRadio`/`jsonArray` (kinds that themselves reject blank). Default error message `"Required"` (matches existing `required` validator); override via `requiredMessage?: string` on BaseField. Closes the latent bypass where a schema author pairs `required: true` with format-only validators (emailNZ/minLength/etc.) that pass through blank input.
- **(2026-06-27)** Phase G: schema-driven pages pass the schema to the client via `<script type="application/json" id="form-schema" set:html={JSON.stringify(schema)}>`, then a module `<script>` imports `mount` from `form-client.ts` and reads it back. JSON roundtrip silently drops `visibleWhen` predicates — safe because `attachVisibleWhen` early-returns when `field.visibleWhen` is undefined. Repeatable add/remove works without any new code because FieldRenderer emits `<template data-repeatable-template>` + `[data-repeatable-row]` + `[data-repeatable-add]` + `[data-repeatable-remove]` already wired by Phase A skeleton.
- **(2026-06-27)** Phase G: defence in depth at the handler boundary. Phase F's implicit `required` doesn't reach nested repeatable item children (walks the leaf path only). The old `checkout-pm.ts` had `isValidPdEntry` filtering — kept that pattern in `coercePdEntries()` on the dynamic handler so malformed nested entries are dropped before they hit the sheet.
- **(2026-06-27)** Phase H: per-entry validation pattern when schema has `repeatable` with itemFields. `walkFields` treats repeatable as a leaf (no descent). The handler builds a synthetic single-entry schema by `flatMap`ing itemFields out of the step's fields, then calls `validate(syntheticSchema, entry)` once per entry. Keeps runtime.ts unchanged; validation logic stays in the handler.
- **(2026-06-27)** Phase I + bug-002: `toRow()` honours `visibleWhen`. Before the fix, hidden fields were written as `""` to the sheet (looked like "user submitted empty"). Fix: early-continue when `field.visibleWhen && !field.visibleWhen(values)`. `validate()` already skipped copy-through for hidden fields; toRow now mirrors that contract.
- **(2026-06-27)** Phase J1: FieldRenderer `inputId = inputName` (was `f-{inputName-with-dots-as-hyphens}`). Top-level field DOM id now equals its name attribute. Rationale: professional/apply.astro hydration used `getElementById("firstName")` — with the old `f-` prefix, ids didn't match names. The dot is valid in HTML5 ids (HTML5 id-attribute spec), so nested fields like `referees.referee1Name` work too. Backward-compat: other forms ([tier].astro, pd-log.astro, apply.astro associate) read via FormData/FormData-based selectors, not getElementById, so the change is invisible to them.
- **(2026-06-27)** Phase J1: render-only delete pattern. When an inline HTML section is replaced by schema-driven render, the corresponding JS that built/manipulated that section (buildCompetencyGrid, addCourseRow, addExperienceRow, the courses/experiences/competencyStates state arrays) becomes dead code. J1 deletes the JS + stubs saveFormData references to the deleted state. J2 will rewrite saveFormData to read FormData(form) instead of in-memory state.
- **(2026-06-27)** Phase L: grid column IDs (and similar lists) derive from `Object.keys(content.steps.{stepId}.fields.{fieldName}.options)` — NOT hardcoded in TS as ELDAA-specific constants. TS owns the PATTERN (a grid of column radios), JSON owns the per-org content (which competencies + which questions). The keys remain the data contract (sheet column order is stable), but they're now editable per-org by rewriting the JSON options map. Same pattern applies to any list-driven field config (e.g. selectField options).
- **(2026-06-27)** Phase J2 + bug fix in `assertOptionValuesExist`: was only checking visibleWhen on select/radio fields' OWN options. Now collects every option value across the schema (including in groups + repeatables) then checks every visibleWhen predicate against the global set. Rationale: a textarea's visibleWhen predicate can reference another field's option values ("show details when listOnPage === 'yes'"). Old behaviour silently let those pass.
- **(2026-06-27)** Phase J2: form-client `attachAutosaveQueue` extended with `serialize` + `contentType` + `onSuccess` options. Default = urlencoded + ignore response. pro apply uses JSON serialize + onSuccess to extract applicantId + resumeLink. Both inline-script (Continue/Back click → explicit `await __flushAutosave__()`) and module-script (debounced background saves) share the same `window.__buildPayload__` helper so the schema-driven payload stays consistent across save paths.
- **(2026-06-27)** Phase J3: upload docTypes come from schema (`professionalApplySchema.uploads.docTypes`), not hardcoded TS arrays. Per-docType descriptions (HTML allowed for the criminal-check link) stay inline in `DOC_DESCRIPTIONS` map keyed by id — schema's docType list doesn't carry HTML, only label + required flag. Rationale: schema is single source of truth for WHAT (label, required), descriptions are styling/content for the UI card. `attachUploadLock` not used here — multi-file queue semantics need cross-button serialisation, which form-client's per-button lock doesn't provide.
